### 3.1 What to ADD

The CRM layer already exists. We only need **one new table** and **one new column**.

#### New Column: Add `site_address` to existing `topics` table

```sql
-- topics is your "projects" table
ALTER TABLE topics ADD COLUMN site_address TEXT;
```

#### New Table: `rfq_workflows`
Tracks the state of each RFQ as it moves through the pipeline.

```sql
CREATE TABLE rfq_workflows (
    id               SERIAL PRIMARY KEY,
    topic_id         INT REFERENCES topics(id),  -- "topic" = your project
    thread_id        INT REFERENCES threads(id), -- ✅ correct table: "threads" (model: Thread)
    status           VARCHAR(100) DEFAULT 'ingested',
    -- Status values: ingested | triage | researching | drafts_ready | sent | quotes_received
    confidence_score FLOAT,       -- from Context Resolver Agent (0.0 to 1.0)
    assigned_by      VARCHAR(100), -- 'ai' or 'human'
    created_at       TIMESTAMP DEFAULT NOW(),
    updated_at       TIMESTAMP DEFAULT NOW()
);
```

#### New Table: `product_results`
Stores market research prices per topic. This table does **not** exist in the current DB — create it in migration 004.

> ⚠️ **Dev note:** If `product_comparisons` already exists in your DB with a compatible structure,
> you may write market research data there instead and skip this CREATE TABLE. Check before running migration.

```sql
CREATE TABLE product_results (
    id           SERIAL PRIMARY KEY,
    topic_id     INT REFERENCES topics(id),
    item_name    TEXT NOT NULL,
    supplier     TEXT,           -- e.g. "Bunnings", "Sydney Tools", "Blackwoods"
    unit_price   NUMERIC(10, 2),
    unit         VARCHAR(50),    -- e.g. "each", "m", "box"
    source_url   TEXT,
    fetched_at   TIMESTAMP DEFAULT NOW()
);
```

### 4.3 New Agent: `agents/context_resolver/`

**Purpose:** Given an incoming email, figure out which topic (project) it belongs to.

**Files to create:**
- `agents/context_resolver/__init__.py` (empty)
- `agents/context_resolver/resolver.py`

**Logic (plain English):**

```
Step 1: Receive sender_email + email_body

Step 2: Search the contacts table
        Query: WHERE sender_email = ANY(contact_emails)
        (contact_emails is a PostgreSQL ARRAY column)

Step 3: If no contact found
        → Return {"status": "triage", "confidence": 0, "reason": "unknown sender"}

Step 4: If contact found
        → Get contact_id
        → Fetch all active Topic records WHERE contact_id = that contact's id

Step 5: If no active topics found
        → Return {"status": "triage", "confidence": 0, "reason": "no active projects for this contact"}

Step 6: Call LLM (Claude API) with the prompt from config/prompts.py
        Pass: email_body + list of topics (id, topic_name, site_address, recent_materials)

Step 7: Parse LLM JSON response
        If confidence >= 0.85 → return {"status": "researching", "topic_id": X, "confidence": 0.92}
        If confidence < 0.85  → return {"status": "triage", "topic_id": X, "confidence": 0.71}
```

**LLM Prompt (goes in `config/prompts.py` as `CONTEXT_RESOLVER_PROMPT`):**

```
You are a procurement project classifier.

Given the email below and the list of active projects for this client,
determine which project this email most likely belongs to.

Email:
{email_body}

Active Projects:
{topic_list}

Respond ONLY in valid JSON — no explanation, no markdown:
{"topic_id": 12, "confidence": 0.92, "reason": "email mentions steel beams matching site address"}
```

Add these two functions **at the bottom** of the existing `tasks.py` (do not touch or remove existing functions):

### 4.6 Email Listener Hook: `integrations/email_listeners/intake_hook.py`

New file. Glues the email listener to the context resolver and background tasks.

**On new email received, this hook does:**

```python
# Pseudocode — implement in intake_hook.py

def on_email_received(sender_email, subject, body, thread_id):

    # 1. Save to existing threads table (already exists — use existing model)
    thread = save_to_threads(sender_email, subject, body, thread_id)

    # 2. Call context resolver
    result = context_resolver.resolve(sender_email, body)

    # 3. Create rfq_workflow record
    workflow = create_rfq_workflow(
        topic_id         = result.get("topic_id"),
        thread_id        = thread.id,
        status           = result["status"],         # "researching" or "triage"
        confidence_score = result["confidence"],
        assigned_by      = "ai"
    )

    # 4. Branch on result
    if result["status"] == "researching":
        # Kick off async background task — does NOT block the webhook
        background_tasks.add_task(run_market_research, workflow.topic_id, parsed_items)

    elif result["status"] == "triage":
        # Do nothing automatically — wait for human assignment on triage.html
        pass
```

## 6. FULL WORKFLOW (End-to-End)

```
1. Email arrives in inbox
        ↓
2. Email Listener (existing) picks it up
   → intake_hook.py saves to threads table
        ↓
3. Context Resolver Agent runs
   → Looks up sender email in contacts.contact_emails (ARRAY)
   → If found: fetches active topics for that contact_id
   → Calls LLM to score which topic this email belongs to
        ↓
   [Confidence ≥ 85%?]
    YES → rfq_workflows record: status = "researching"
    NO  → rfq_workflows record: status = "triage"
          (appears on triage.html — human assigns manually)
        ↓
4. Background Task: run_market_research()
   (async — does NOT block webhook)
   Searches Bunnings, Sydney Tools, Blackwoods for prices
   Saves to product_results table
        ↓
5. Background Task: trigger_vendor_drafting()
   vendor_matcher finds right suppliers from suppliers table
   rfq_agent (existing) drafts emails
   Saved to rfqs table as DRAFT — NOT sent
        ↓
6. Human reviews drafts in rfq.html
   Progress stepper shows current stage
   Human approves → status = "Sent" → emails go out
        ↓
7. Supplier replies → saved to supplier_quotes
        ↓
8. Progress stepper on rfq.html updates to "Quotes Received" ✅
```
